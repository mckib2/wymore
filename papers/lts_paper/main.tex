\miniheading{Abstract}
This working draft serves as a collaborative workspace to establish the outline, draft content, and key discussion prompts for Paper 1. We propose a formal systems engineering pipeline that bridges requirements to physical realization by introducing a three-tiered Buildability Cotyledon refinement: Conceptual, Logical, and Physical. The Conceptual layer models abstract concurrent behavior as a Labeled Transition System (LTS), which is verified against Linear Temporal Logic (LTL) specifications. The transition to the Logical layer—modeled as Wymore's deterministic 5-tuple discrete system—is validated using weak bisimulation under the Synchrony Hypothesis. Finally, the Platform layer refines the logic into resource-constrained Colored Petri Nets (CPNs), establishing end-to-end mathematical traceability.

\phantomsection
\subsubsection{Keywords}
Model-Based Systems Engineering (MBSE), Labeled Transition Systems (LTS), Tricotyledon Theory of System Design, Linear Temporal Logic (LTL), Weak Bisimulation, Cyber-Physical Systems (CPS).

\section{Results: Criteria for a Candidate Case Study}
Rather than committing to a specific target system, this section outlines the required properties of a candidate case study to validate our proposed three-tiered refinement pipeline (LTS $\rightarrow$ Wymore $\rightarrow$ CPN). A strong case study must exhibit the following attributes:

\miniheading{1. Concurrent Action-to-I/O Refinement:}
The system must start with symmetric concurrent interactions (e.g. handshake protocols) at the Conceptual LTS level, which are refined into partitioned Inputs and Outputs at the Logical Wymorean level. This validates the bisimulation bridge theorem and shows how asymmetric interfaces can be derived formally.

\miniheading{2. Tick-Based Synchrony Mapping:}
The case study must transition from event-driven conceptual transitions to time-driven, clock-ticked logical trajectories. This allows demonstrating how to align time ticks with event intervals (under the Synchrony Hypothesis) and verifying that the bisimulation holds under this alignment.

\miniheading{3. Resource Constraint Mapping (CPN Integration):}
The logical 5-tuple must be mapped to physical constraints at the Platform level (e.g., bus transmission delays, CPU execution times, or battery/power constraints). Representing these physical attributes as Colored Petri Net (CPN) tokens demonstrates how functional requirements are preserved under hardware limitations.

\miniheading{4. Decoupled Verification Boundaries:}
The case study should feature requirements that can be checked independently at the abstract level ($G(\text{Req} \rightarrow F \text{Ack})$) so that changing physical allocation (e.g., changing bus baud rates or processor core speeds) only requires re-verifying the local platform mapping, rather than re-proving the whole system.


\section{Discussion}
This section will critically analyze the theoretical and practical implications.

\miniheading{Key points to expand upon:}
\begin{itemize}
    \item \textbf{Decoupling of Intent and Structure:} Changing state variable layouts or implementation protocols does not break verification, as only the bisimulation mapping needs to be re-evaluated.
    \item \textbf{The Synchrony Hypothesis:} Wymore's 5-tuple operates on time-driven trajectories (ticking clocks), whereas the LTS is event-driven. We need a rigorous discussion on standardizing time ticks to bridge these paradigms \parencite{LynchTuttle1987}.
\end{itemize}

\miniheading{Abstract}
We propose an assertional reinterpretation of the Functionality Cotyledon in A.\ Wayne Wymore's Tricotyledon Theory of System Design (T3SD). In classical T3SD, functional intent is verified constructively between a reference system $Z_{\text{spec}}$ and implementation $Z_{\text{impl}}$ via homomorphism $h: Z_{\text{impl}} \rightarrow Z_{\text{spec}}$. We offer an assertional reading of the FC as $\mathcal{FC}_{\mathcal{L}}(\Phi) = \{ Z \mid Z \models_{\mathcal{L}} \Phi \}$, where $\Phi$ is a set of temporal logic properties over Wymore discrete-system trajectories and $\mathcal{L}$ is a chosen temporal logic dialect. The contribution is a reinterpretation of the elements of the Functionality Cotyledon, trace semantics, and workflow comparison. We illustrate the comparison on a decentralized UAV swarm with infinite product state and dynamic system configuration.

Tricotyledon Theory of System Design (T3SD), Linear Temporal Logic (LTL), Formal Verification, Model-Based Systems Engineering (MBSE).

\section{Introduction}

Model-Based Systems Engineering (MBSE) promises improved traceability and earlier verification, yet many requirements remain in natural language or diagrammatic forms without checkable temporal semantics \parencite{salado2019mbse}. Validation then depends on manual reconstruction of intent: engineers cannot mechanically determine whether an implementation satisfies any given requirement without informal argument. Encoding requirements as temporal properties over explicit system trajectories is one path toward model-based requirements that are executable rather than purely documentation \parencite{salado2019mbse}.

A.\ Wayne Wymore's Tricotyledon Theory of System Design (T3SD) offers a rigorous foundation for this gap: cotyledon-structured design spaces, trajectory theorems, and constructive verification via reference systems and homomorphisms \parencite{Wymore1993,salado2019wymore}. Generic model checking lacks T3SD's design-space semantics; we reinterpret the Functionality Cotyledon (FC) assertionally while preserving cotyledon structure. Constructing and maintaining homomorphisms is largely manual and poorly matched to evolving implementations, especially when concurrent subsystems induce large proof obligations \parencite{Wymore1993}.

We propose encoding functional requirements as a property set $\Phi$ in temporal logic dialect $\mathcal{L}$ and verifying implementations by checking $Z \models_{\mathcal{L}} \Phi$ over admissible Wymore trajectories. In contrast to structural morphisms, assertional FC does not require a concrete $Z_{\text{spec}}$ for checking.

\miniheading{Contributions}
\begin{itemize}
    \item A parameterized assertional reinterpretation of the Functionality Cotyledon, reframing functional verification as temporal property checking over Wymore trajectories rather than as construction of system morphisms.
    \item Trace-based verification semantics for state and output trajectories, aligned with Wymore's trajectory theorems \parencite{Wymore1993}, supporting model-based requirements executable against open-system boundaries \parencite{salado2019mbse}.
    \item A workflow comparison of constructive and assertional verification methods on a dynamically configured UAV swarm system.
\end{itemize}

The paper proceeds as follows: Section \ref{sec:background} summarizes T3SD, notation, and temporal logics; Section \ref{sec:methodology} defines trace semantics and verification workflows; Section \ref{sec:case-study} instantiates a UAV swarm system and compares constructive and assertional workflows; Section \ref{sec:discussion} relates assertional FC to classical T3SD.

\section{Background} \label{sec:background}

\subsection{Tricotyledon Theory of System Design}
Wymore's T3SD organizes system design into three cotyledons (design spaces): Functionality (FC), Buildability, and Implementability \parencite{Wymore1993}. The FC contains system designs that satisfy required input/output behavior and preferred performance ordering. Buildability captures technology constraints and cost orderings while Implementability is the space of designs that homomorphically map buildable artifacts to functional ones. In classical verification, functional intent is realized constructively: system requirements are formalized into a reference system $Z_{\text{spec}}$, and implementations are checked via morphisms. This paper focuses specifically on the FC.

Figure~\ref{fig:t3sd-cotyledons} situates both verification readings within the cotyledon layout: $Z_{\text{spec}}$ and property set $\Phi$ formalize functional intent in the FC, while $Z_{\text{impl}}$ resides in the buildability space; constructive and assertional checks cross the implementability intersection via homomorphism $h$ or trace satisfaction $Z_{\text{impl}} \models_{\mathcal{L}} \Phi$.

We restrict attention to the subset of system designs that can be expressed within Wymore's discrete systems (DSYSTEMS): a deterministic 5-tuple $Z = (S, I, O, \delta, \rho)$ with state set $S$, input set $I$, output set $O$, transition function $\delta$, and readout $\rho$ \parencite{Wymore1993}. Time index $t \in T$ advances in discrete, atomic ticks while state components may be continuous-valued at each tick (e.g., $x,y \in \mathbb{R}$).  ``Discrete system'' here refers to the clock, not to finite or integer-valued states. Verification in this paper evaluates properties at tick boundaries. Wymore explicitly allows $S$, $I$, and $O$ to be \emph{not-necessarily-finite}, i.e., possibly infinite, compared to finite state machines only. Given an input trajectory $f: T \to I$ and initial state $s_0 \in S$, Wymore's trajectory theorems yield unique state and output trajectories $g: T \to S$ and $y: T \to O$. Systems of interest here are open: verification is performed at the system boundary through observable inputs and outputs, consistent with Systems Theory \parencite{salado2019mbse}.

\subsection{Temporal Logics}
Temporal logics are formal languages for stating requirements over time-indexed execution traces \parencite{Pnueli1977,BaierKatoen2008}. They are used to specify safety (forbidden conditions never occur), liveness (desired conditions eventually occur), and ordering constraints on behavior.

\renewcommand{\arraystretch}{1.35}
\footnotesize

    \rowcolor{tableheader}
    \headingfont\bfseries Op. & \headingfont\bfseries Name & \headingfont\bfseries Example & \headingfont\bfseries Reading (over trace $\sigma$) \\
    \addlinespace[2pt]
    $G\,\varphi$ & always & $G\,\text{safe}$ & $\varphi$ holds at every tick \\

    $X\,\varphi$ & next & $X\,\texttt{ack}$ & $\varphi$ holds at the next tick \\
    $\varphi \rightarrow \psi$ & implication & $\text{start} \rightarrow F\,\text{complete}$ & if $\varphi$ holds, then $\psi$ eventually holds \\
\end{tabularx}
\captionof{table}{Common LTL operators and illustrative readings.}
\label{tab:ltl-operators}
\endgroup
\vspace{4pt}

Table~\ref{tab:ltl-operators} summarizes the operators used below. First-Order LTL (FO-LTL) adds quantifiers over state, input, or output variables at each tick which are needed when requirements relate continuous variables or indexed elements. Other dialects (e.g., Signal Temporal Logic (STL) for continuous signals) remain admissible as $\mathcal{L}$. Because functional intent varies in complexity from discrete event sequencing to continuous signal bounds, our framework parameterizes the chosen logic as $\mathcal{L}$ which allows arbitrarily complex property specification.

\subsection{Related Formalisms}
Generic state model checking does not, by itself, supply Wymore trajectory semantics or a constructive-versus-assertional account of T3SD verification. Hybrid and event-scheduled formalisms address dynamics outside Wymore's discrete systems with uniform, atomic clock \parencite{platzer2018logical,wach2021devs}.

\section{Methodology} \label{sec:methodology}

\subsection{Verification Paradigm Comparison}
Figure~\ref{fig:comparison} instantiates the verification workflow sketched in Figure~\ref{fig:t3sd-cotyledons}: in both panels, system requirements are formalized as a checkable artifact ($Z_{\text{spec}}$ or $\Phi$), and $Z_{\text{impl}}$ is verified at the system boundary via morphism $h$ or property checking. Table~\ref{tab:comparison} summarizes the generic comparison.  When implementations change structure, maintaining universal constructive proofs over $Z_{\text{spec}}$ and $h$ can require substantial re-verification. Assertional checking instead evaluates whether the post-change system satisfies $\Phi$ on its trajectories, without necessitating the construction of a new reference system.

\subsection{Formal Trace Definitions and Observability}
\label{sec:trace-observability}
For $Z = (S, I, O, \delta, \rho)$ and trajectories $(f, g, y)$ as in Section~\ref{sec:background}, we write $g(t)$ for state and $y(t)$ for output at tick $t \in T$. Wymore's standard trajectory construction is:
\begin{align}
    g(0) &= s_0 \\
    g(t+1) &= \delta(g(t), f(t)) \\
    y(t) &= \rho(g(t))
\end{align}
We define the complete execution trace as $\sigma = (g(t), y(t))_{t \in T}$. Temporal requirements may use mixed observability:
\begin{itemize}
    \item External (black-box): $\Phi_{\text{ext}}$ over outputs $y(t)$ and inputs $f(t)$ only.
    \item Internal (white-box): properties over state components in $g(t)$.
\end{itemize}

Under the assertional workflow, systems engineers (1)~elicit boundary requirements, (2)~encode $\Phi$ in dialect $\mathcal{L}$ (optionally splitting $\Phi_{\text{ext}}$ and internal properties), (3)~obtain $Z_{\text{impl}}$ (possibly via some know system coupling recipe), and (4)~check $Z_{\text{impl}} \models_{\mathcal{L}} \Phi$ under a stated admissible input class. Admissible input trajectories $f$ are those consistent with mission protocol and environment assumptions at the system boundary, not arbitrary elements of $I^T$ unless explicitly required \parencite{salado2019mbse}. Port-level liveness and coordination belong in $\Phi_{\text{ext}}$; physical invariants not exported at ports are checked over $g$. When both $Z_{\text{spec}}$ and $\Phi$ are present, $\Phi$ may abstract the same mission intent at ports while $Z_{\text{spec}}$ supports constructive $h$; assertional checking does not require $Z_{\text{spec}}$. Properties over multiple agents are evaluated on the resultant system trajectory, not per-agent in isolation.

We say $Z \models_{\mathcal{L}} \Phi$ if, for all admissible input trajectories $f$, the resulting trace satisfies $\sigma \models_{\mathcal{L}} \Phi$ under dialect $\mathcal{L}$ \parencite{Pnueli1977,BaierKatoen2008}. We write $\mathcal{FC}_{\mathcal{L}}(\Phi) = \{ Z \mid Z \models_{\mathcal{L}} \Phi \}$, where dialect $\mathcal{L}$ is chosen to match requirement complexity. When $\mathcal{L}$ is clear we write simply $Z \models \Phi$. Verification is scoped to Wymore DSYSTEMS at discrete ticks; $S$, $I$, and $O$ may be infinite, and checking strategy is tool-dependent \parencite{Wymore1993,BaierKatoen2008}. Subsystems may advance via internal $\delta$ when no external command arrives (i.e., autonomous state transition).  We treat such steps as ticks with hold-last-input or environment-defined $f(t)$. Assertional and classical FC need not coincide: existence of a homomorphism $h$ guarantees that every implementation trajectory maps to a valid specification trajectory (structural simulation), whereas $Z \models_{\mathcal{L}} \Phi$ judges only the trace variables referenced in $\Phi$. Two implementations may therefore satisfy the same black-box properties while differing in unobserved state (under-specification relative to $h$), and a liveness clause may demand progress not encoded in a given $Z_{\text{spec}}$ transition structure (over-specification relative to $h$).

\subsection{Constructive Verification Criteria}
\label{sec:constructive-criteria}
Classical FC seeks a system homomorphism $h : Z_{\text{impl}} \rightarrow Z_{\text{spec}}$: surjective maps on states, inputs, and outputs that preserve $\delta$ and $\rho$ \parencite{Wymore1993}. Intuitively, every trajectory of $Z_{\text{impl}}$ is simulated by a corresponding trajectory in $Z_{\text{spec}}$. When $Z_{\text{impl}}$ is built from concurrent subsystems, Wymore's conjunctive composition (CSY) forms a resultant system with shared input ports and per-component readouts \parencite{Wymore1993}; the homomorphism obligation then relates the composed implementation to a (typically simpler) reference system.

\section{Case Study: UAV Swarm}
\label{sec:case-study}

We apply the proposed methodology to an example system in order to compare constructive and assertional workflows on the same $Z_{\text{impl}}$.  This is done as a conceptual demonstration, not an empirical benchmark.

\subsection{Specification System Definition}
\label{sec:swarm-spec}

Under classical T3SD, swarm functional intent is first expressed as a behavioral reference system $Z_{\text{spec}}$ (Table~\ref{tab:swarm-quintuples}, left column; Figure~\ref{fig:swarm-verification}). States record aggregate search progress $(K,t,a)$: covered cells $K \subseteq \text{Cells}$ over region $R$, mission clock $t$, and phase $a$. Boundary ports are broadcast \texttt{MissionCmd} and aggregate \texttt{MissionOut}; $\rho$ maps $(K,t,a)$ to \texttt{coveragePct}, \texttt{missionActive}, \texttt{deadlineMet}, and \texttt{ack}. On \texttt{tick}, $t$ advances and $K$ grows when active agents enter new cells; liveness requires $K = \text{Cells}$ before $t > T_{\max}$ while $a = \text{searching}$.

\subsection{Implementation System Definition}
A decentralized swarm of $N$ UAVs executes cooperative search over a bounded region $R \subset \mathbb{R}^2$ while maintaining minimum separation $d_{\text{safe}}$. A Return to Base (RTB) event removes a vehicle from the current \emph{active set} $M \subseteq \{1,\ldots,N\}$ when its battery falls below threshold. Participation change is the deliberate confounder: for constructive proofs, dynamic system configuration may require revisiting a homomorphism $h$ between the specification and current system configuration, whereas assertional checking re-evaluates the same $\Phi$ on the suffix trace.

The implementation $Z_{\text{impl}}$ is a conjunctive composition (CSY) of $N$ agent subsystems sharing \texttt{MissionCmd} at the mission bus (Table~\ref{tab:swarm-quintuples}, right column; Figure~\ref{fig:swarm-ov1}). Assertional specification $\Phi = \{\varphi_{\text{safe}}, \varphi_{\text{live}}\}$ in FO-LTL provides the alternative functional artifact; port and property detail appear in Section~\ref{sec:artifact-definitions}.

\subsection{Artifact Definitions}
\label{sec:artifact-definitions}

\miniheading{Mission Port Vocabulary}
Boundary types are shared by $Z_{\text{impl}}$, $Z_{\text{spec}}$, and black-box checking of $\Phi$ (Figure~\ref{fig:swarm-verification} summarizes the wiring).

\textbf{Input} \texttt{MissionCmd} (broadcast on one mission bus): \texttt{startSearch} sets phase $a \leftarrow \text{searching}$; \texttt{pause} holds the mission clock and sets $a \leftarrow \text{idle}$; \texttt{abort} terminates with $a \leftarrow \text{aborted}$; \texttt{tick} advances clock $t$ by one flight-controller cycle (default when no operator command is present).

\textbf{Aggregate output} \texttt{MissionOut} (swarm-level $\rho$ on both systems): \texttt{coveragePct} $\in \{0,\ldots,100\}$ is $100|K|/|\text{Cells}|$ capped at 100; \texttt{missionActive} is true iff $a = \text{searching}$; \texttt{deadlineMet} is true iff $t \leq T_{\max}$; \texttt{ack} is true iff $a \neq \text{aborted}$.

\textbf{Per-agent output} \texttt{AgentStatus} (one CSY status port per $Z_i$, implementation only): \texttt{active} is true iff agent $i \in M$; \texttt{rtbPending} is true iff $b_i$ is below the RTB threshold.

Mission phase $a \in \{\text{idle}, \text{searching}, \text{complete}, \text{aborted}\}$. In $\varphi_{\text{live}}$, $\text{Command}_{\text{search}}$ denotes issuance of \texttt{startSearch} on the mission bus.

\miniheading{Assertional Specification $\Phi$ (FO-LTL)}
We instantiate dialect $\mathcal{L}$ as FO-LTL. $\varphi_{\text{safe}}$ uses internal system state in $g$ (inter-agent separation over active pairs in $M$); $\varphi_{\text{live}}$ uses only mission outputs $y$ (\texttt{coveragePct} on \texttt{MissionOut}). Functional intent:
\begin{align*}
\varphi_{\text{safe}} &= G \bigl( \forall i,j \in M,\, i \neq j \rightarrow d_{ij} \geq d_{\text{safe}} \bigr) \\
\varphi_{\text{live}} &= G \bigl( \text{Command}_{\text{search}} \rightarrow F\, (\texttt{coveragePct} = 100) \bigr)
\end{align*}
where $d_{ij}$ is Euclidean distance between agents $i$ and $j$, and $M$ is the active-agent set.

\miniheading{Constructive Reference $Z_{\text{spec}}$}
As defined in Section~\ref{sec:swarm-spec}, $Z_{\text{spec}}$ abstracts continuous swarm motion to aggregate cell coverage $(K,t,a)$ at shared mission ports; $\varphi_{\text{live}}$ checks the same \texttt{MissionOut} fields that $\rho$ exposes on $Z_{\text{spec}}$.

\miniheading{Illustrative Instance: RTB ($N=10 \rightarrow 9$)}
Suppose UAV$_3$ enters RTB; agent~3 leaves $M$ while nine agents continue. The implementation's system tuple changes: $S_{\text{swarm}}$ is a product over $N$ agents and $\delta$ encodes RTB, so the post-event system is not the same mathematical object for which $h$ was proved. Unless a universal active set invariant was established upfront, the surjective state map and $\delta$-preservation obligations of $h$ do not automatically survive and system engineers must now construct a new homomorphism proof. For the proposed assertional methodology, we must only re-check the same $\Phi$ on the suffix trace with updated $M$.  If $\varphi_{\text{safe}}$ and $\varphi_{\text{live}}$ hold, $Z_{\text{impl}} \models_{\mathcal{L}} \Phi$ without rebuilding $Z_{\text{spec}}$.

This case study exercises an infinite state space, continuous coordinates at discrete ticks, FO-LTL with mixed observability, and participation change as proof-maintenance for $h$, not as impossibility of classical T3SD.  Achieving dynamic system reconfiguration is possible under classical T3SD, but we expect it to be much more expensive than the assertional methodology.

\section{Discussion} \label{sec:discussion}

Encoding functional requirements as temporal logic moves intent from natural-language statements into a checkable formal language \parencite{salado2019mbse,BaierKatoen2008}, enabling explicit semantics for requirements and their execution/analysis against system trajectories. Assertional trace checking aligns with open-system validation in systems engineering practice \parencite{salado2019mbse}, though we do not claim normative alignment with any particular process. %When $\delta$ or $\rho$ change, engineers may re-check $Z \models_{\mathcal{L}} \Phi$ rather than rebuilding homomorphisms.

T3SD pedagogy traditionally introduces constructive FC first: students formalize $Z_{\text{spec}}$ and prove $h$ \parencite{Wymore1993}. Assertional FC is offered here as a reinterpretation of the same Functionality Cotyledon for verification workflow, not a replacement of its structure or of classical T3SD when a reference system already exists. We suggest that Wymore's allowance for not-necessarily-finite state and trajectory-based semantics supports evaluation of temporal logic at ticks as a dual reading of the same design space.

\subsection{Relationship to Classical Functionality Cotyledon}
\label{sec:classical-fc}

Classical T3SD defines functional designs constructively while assertional FC defines them by temporal properties in dialect $\mathcal{L}$. Morphism existence implies trace inclusion under the homomorphic map; conversely, $Z \models_{\mathcal{L}} \Phi$ does not imply existence of $h$ to a given $Z_{\text{spec}}$ (Section~\ref{sec:trace-observability}). Logical equivalence between these methods is left to future work.

We also note that constructive verification may still be preferable when $|S|$ is small, $Z_{\text{spec}}$ already exists, and changes are internal refinements with fixed observable behavior.

\subsection{Expressivity and Tooling}
\label{sec:expressivity}

Assertional FC is intended to capture functional intent through $\mathcal{FC}_{\mathcal{L}}(\Phi)$ without requiring a reference system definition. At the theoretical level, $\mathcal{L}$ may range from propositional LTL to FO-LTL over trajectory variables at ticks; the UAV case study requires first-order logic quantifiers over agents and continuous coordinates, which presents significant computational challenges. In practice, tooling may restrict $\mathcal{L}$ to decidable fragments (finite-state model checking, tick-local SMT).  E.g., propositional LTL alone is a \emph{tooling subset} that may be sufficient for many requirement specifications, especially those that may have otherwise been written against finite-state models.

\subsection{Limitations and Open Questions}
\label{sec:limitations}

\begin{itemize}
    \item \textbf{Equivalence:} We have not demonstrated that assertional and classical FC are equivalent; Tables~\ref{tab:comparison}--\ref{tab:swarm-verification} compare workflows, not logical strength (see also Section~\ref{sec:expressivity}).  We believe that arbitrarily complex requirements may always be expressed using some temporal logic, but this is left to future work.
    \item \textbf{Theory vs.\ tooling:} FO-LTL as used in the case study over real-valued coordinates is undecidable in general; practitioners may restrict $\mathcal{L}$ for tractable checking.
    \item \textbf{Infinite state:} In the case study, the swarm system may have been modeled to use tick-local SMT to check separation at sampled states, an under-approximation for inter-tick motion \parencite{BaierKatoen2008}.  Continuous-time between ticks is left to future work.
    \item \textbf{Constructive tractability:} Universal proofs over all participation masks and real trajectories remain open.
    \item \textbf{Autonomous steps:} Complete liveness semantics for ticks without external input is an area for future theoretical work.
    \item \textbf{Empirical / MBSE evidence:} We provide no benchmark comparing morphism effort to property checking and no SysML requirements trace in this paper. Integration with model-based requirements workflows and standard tooling is left to future work.
\end{itemize}

\section{Conclusions}

In this paper, we offer an assertional reinterpretation of Wymore's Functionality Cotyledon: functional verification as temporal property checking over trajectories, with trace semantics aligned to T3SD. The UAV swarm case study illustrates mixed observability, infinite product state, and RTB participation change, where assertional re-checking of $\Phi$ avoids rebuilding $Z_{\text{spec}}$ and $h$. Logical equivalence between paradigms, autonomous-step semantics, and MBSE tool integration remain open.

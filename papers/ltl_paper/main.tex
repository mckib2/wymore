\miniheading{Abstract}
We focus on modernizing Wayne Wymore's Tricotyledon Theory of System Design (T3SD), specifically reinterpreting the Functionality Cotyledon as a declarative set of Linear Temporal Logic (LTL) and First-Order LTL (FO-LTL) formulas. Rather than constructing manual, rigid system morphisms between a specification and implementation, we propose verifying Wymore's deterministic 5-tuple models directly against desired temporal requirements, captured as LTL expressions.

Tricotyledon Theory of System Design (T3SD), Linear Temporal Logic (LTL), Formal Verification.

\subsection{Formal Trace Definitions and Observability}
Given a Wymorean system $Z = (S, I, O, \delta, \rho)$ and an input trajectory $f \in \text{ITZ}$, the system generates unique state and output trajectories:
\begin{align}
    g(0) &= s_0 \\
    g(t+1) &= \delta(g(t), f(t)) \\
    h(t) &= \rho(g(t))
\end{align}
We define the complete execution trace as the sequence $\sigma = (g(0), h(0)), (g(1), h(1)), \dots$. We consider our temporal requirements $\Phi$ based on observability constraints:
\begin{itemize}
    \item \textbf{External Observability (Black-Box):} Properties $\Phi_{\text{ext}}$ are defined only over output trajectories $h(t)$ and inputs $f(t)$. This represents standard system validation where internal states are hidden.
    % TODO: cite Salado work on system theoretic definitions of closed and open system -- we are in the problem space of solutions, so we need only open systems
\end{itemize}
We say $Z \models \Phi$ if for all valid input trajectories $f$, the resulting trace satisfies $\sigma \models \Phi$ under standard LTL semantics \parencite{Pnueli1977}.

\subsection{Continuous Dynamics and Decidability Bounds}
Evaluating LTL over continuous states and trajectories (e.g., using FO-LTL to check properties like $G \forall x (\text{Input}(x) \rightarrow F \text{Output} = f(x))$) introduces undecidability. % NBPM: why must we consider continuous states?  Wymorean systems are explicitly in DSYSTEMS, not continuous.  Can be extended to DEVS if we need this (see T3SD DEVS paper)

To maintain decidability, we frame our methodology as evaluating trajectories at discrete time units and leveraging:
\begin{enumerate}
    \item **Bounded Model Checking (BMC):** Restricting verification to finite prefixes of the trajectories.
    \item **SMT-Backed Invariant Checking:** Interfacing with solvers (like Z3) to evaluate continuous variables at discrete steps.
\end{enumerate}
This contrasts with continuous-time hybrid systems verification models, such as Differential Dynamic Logic (dL) verified in provers like KeYmaera X \parencite{platzer2018logical}, by constraining continuous reasoning to static algebraic assertions at discrete ticks.  % NBPM: Need to also relate to DEVS -- that's probably the most important comparison in the existing literature

\section{Results}
A strong case study must exhibit the following attributes:

\miniheading{1. State-Space Complexity and Concurrency:}
The system must have enough concurrent state components (e.g. sensor subsystems, mode changes, user overrides) that composition leads to a state-space explosion. This makes drawing a manual reference specification $Z_{\text{spec}}$ or finding an explicit morphism $h: Z_{\text{impl}} \rightarrow Z_{\text{spec}}$ mathematically untenable, highlighting the need for automated trace checking.

\miniheading{2. Continuous-Discrete Trajectory Interface:}  % NBPM: why continuous?  Can we get away with discrete?
The system must couple discrete logic with continuous inputs and outputs (e.g. tracking physical distance or velocity over time). This allows demonstrating how SMT solvers or bounded model checking can handle real-valued trajectories at discrete Wymorean clock ticks, proving the decidability limits of the proposed approach.

\miniheading{3. Expressive Safety and Liveness Requirements:}
The requirements must include both safety invariants ($G \neg \text{Hazard}$) and liveness/responsiveness properties (e.g., $G(\text{Trigger} \rightarrow F \text{Response})$). This shows how LTL and FO-LTL capture complex temporal behavior that static structural specifications cannot easily express.

\miniheading{4. Morphism-Defying Edge Cases:}
The case study should feature edge cases (e.g., transient sensor failures or unexpected environment cut-ins) where the implementation system's state space changes structurally. Under classical T3SD, this requires rebuilding the entire constructive reference system and morphism. Under our assertional framework, it only requires adding or modifying a temporal assertion.

\section{Discussion}

\begin{itemize}
    \item \textbf{Decoupled Intent and Structure:} Shows how modifying state variables or output mappings doesn't require rebuilding structural morphisms, only re-running the model checker.
    \item \textbf{Decidability of FO-LTL:} Address the complexity limits when continuous values are checked over trajectories.
\end{itemize}

\section{Conclusionss}

The paper redefines Wymore's Functionality Cotyledon assertionally as $\mathcal{FC}(\Phi) = \{ Z \mid Z \models \Phi \}$. This replaces constructive structural morphisms with direct temporal logic verification, enabling automated model checking of systems engineering specifications.
